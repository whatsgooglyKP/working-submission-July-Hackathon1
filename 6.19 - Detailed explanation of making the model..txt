  ### 2. Rationale (Why We Did It)

  • Input Validation & UX Safeguard: Restricting the button execution ensures that the model is never invoked with empty parameters. It prevents API failure logs and guides the user cleanly on what input
  format is expected (e.g., standard  linkedin.com  link).
  • Consolidated Schema Mapping: Adding  tailored_resume  to  ApplicationStrategy  guarantees that the generated resume conforms to a robust structural template, avoiding loose output formatting bugs.
  • Improved Presentation: Utilizing  st.code  within a collapsible expander creates a dedicated user interface that lets candidates copy their new resume layout instantly, maintaining design consistency
  with the Cover Letter box.

  ### 3. Potential Drawbacks & Mitigations

  #### A. Security Impact

  • Exposure of Profile PII:
      • Risk: Users entering active LinkedIn URLs and uploading resumes might transmit sensitive Personally Identifiable Information (PII), such as phone numbers, email addresses, and employment history to
      the model API endpoints.
      • Mitigation: The application communicates using secure, encrypted HTTPS endpoints to Google GenAI. No profile metadata is cached on permanent storage or shared publicly; the session caches only exist
      in ephemeral RAM.
  • URL/Prompt Injection:
      • Risk: A malicious user could input an injection script inside the LinkedIn URL field or custom notes to hijack model instructions.
      • Mitigation: Pydantic schema validation forces structured JSON outputs, ensuring that even if there is an injection attempt, the output structure remains restricted to the specified fields.


  #### B. Token Burn Impact

  • Increased Output Payload:
      • Risk: Rewriting an entire resume generates a significantly larger token output (approximately 300 to 700 additional output tokens per generation request). Output tokens are more expensive than input
      tokens, and this increase causes slightly higher latency.
      • Mitigation: The app uses a state-aware composite caching layer ( st.session_state.cached_strategies ) that checks if the same combination of inputs has already been generated. If the user clicks
      optimization multiple times with the same LinkedIn URL, notes, and job, the cached strategy is returned, causing zero repeated token burn.
